01 — Overview

Introduction

GhostNet is a Linux-based free and open source encrypted communications platform built on the Reticulum Network Stack (RNS) and LXMF messaging protocol. It provides encrypted, infrastructure-free, decentralized peer-to-peer messaging across mesh, radio, and TCP networks.

GhostNet is intended to give operators a straightforward means of communication across a local or wide-area mesh — whether over LAN, the internet, LoRa radio, or HF/VHF/UHF soundcard modems. No central server, no subscription, no registration.

⚠ Intended Use This software is intended to allow the "average joe" a means of communication for recreation, experimentation or in times of emergency. Because reticulum is encrypted by default, mind your local laws when deploying across radio networks. This may be illegal depending on your area in the world.
02 — Setup

Installation

Online Install (recommended)

Use this method if the target machine has an internet connection. Open a terminal and navigate to the folder where you downloaded the .deb file, then run: 

# Navigate to the folder containing the .deb file
cd ~/Downloads

# Install with apt (replace the filename if your version differs)
sudo apt install ./meridian-ghostnet_1.3.3_amd64.deb

apt will automatically download and install any missing system dependencies, then set up and start the GhostNet service. You only need to run this once.

Offline / Airgap Install

Use this method if the target machine has no internet access. The .tar.gz bundle includes all required system packages and a universal installer — no internet connection is needed on the target machine.

  1. Transfer the file to the target machine — copy it to a USB drive, or use scp if you have SSH access. The file to transfer is:
    meridian-ghostnet_1.3.3_amd64.tar.gz
  2. Open a terminal on the target machine and navigate to where you copied the file. If you copied it to a USB drive mounted at /media/yourname/USB, for example:
    cd /media/yourname/USB
    Or if you copied it to your home folder:
    cd ~/
  3. Extract the archive:
    tar -xzf meridian-ghostnet_1.3.3_amd64.tar.gz
    This creates a new folder called meridian-ghostnet_1.3.3_tarball.
  4. Navigate into the new folder:
    cd meridian-ghostnet_1.3.3_tarball
  5. Run the installer:
    sudo bash install.sh
    The installer will set up all dependencies from the bundled packages, then start the GhostNet service automatically.
ℹ After Installation Open a web browser on the same machine and go to http://localhost:8090. From any other computer on the same network, use your station's IP address: http://<your-ip>:8090
03 — First Launch

Basic Station Configuration

Upon deployment, the operator is presented with the main interface consisting of two primary panels: Station Configuration and Contacts.

Station Configuration is where you set your RNS identity, display name, and configure which network interfaces this node will use. By default, a LAN (AutoInterface) is active — all GhostNet nodes on the same subnet and network group automatically discover each other and form a local mesh.

ℹ Full Router Mode When multiple interfaces are configured (e.g. LAN + LoRa radio), the node operates as a full router — all interfaces are bridged into a single mesh and traffic is forwarded between them automatically.
04 — Networking

Interfaces

GhostNet supports several interface types that can be added from the Node → Interfaces panel. Each interface connects to a different transport layer and all active interfaces are bridged into one unified mesh.

LAN / AutoInterface

Zero-config local mesh. All nodes on the same subnet and network group discover each other automatically. No settings required.

TCP Client

Connect to a known RNS hub by IP address or FQDN and port. Use to bridge to a remote network or a public Reticulum gateway.

TCP Server

Accept inbound Reticulum connections. Makes this node a hub that other TCP clients can connect to.

KISS Radio Modem

Soundcard-based packet radio using Dire Wolf as the software TNC. Works on VHF, UHF, or HF. The following PTT methods were confirmed working during lab testing:

  • DigiRig Lite — PTT: VOX
  • DigiRig Mobile — PTT: RTS
  • Yaesu SCU-17 — PTT: DTR
RNode (LoRa)

LoRa radio module running RNode firmware (LilyGO, Heltec, RAK, etc.). Long range, low power, no license required in most regions.

RNS Propagation Node

Turns this station into a store-and-forward relay. Stores messages for offline nodes and delivers them when they reconnect. Requires a stable, always-on machine.

⚠ RF Interfaces All RF interfaces (KISS Modem, RNode) automatically apply announce_cap = 10% to limit airtime spent on routing announcements. This prevents flooding slow links. The bitrate is set automatically based on the band you select in the wizard.
05 — Discovery

Contacts & Phone Book

The Contacts panel has two tabs:

  • Network — nodes discovered live on the mesh. Any station that has announced within range appears here automatically.
  • Phone Book — your saved contacts. You can store a display name and notes against any destination hash.

Contacts are identified by their destination hash — a cryptographic address derived from their RNS identity. You cannot impersonate another node. All messages are end-to-end encrypted using RNS's built-in cryptography.

06 — Topology

Basic Point-to-Point Network

The simplest GhostNet deployment is two nodes — each with any interface type in common. All traffic on the link is encrypted end-to-end.

Point-to-Point · Two-Node Example

Any pair of stations with a shared interface type can communicate directly. The link can be LAN, TCP over the internet, LoRa radio, or a soundcard packet modem — the application layer is identical regardless of transport.

07 — Topology

Robust Mesh Network Topology

For broader coverage, GhostNet nodes automatically route traffic across multiple hops. Any node configured as a full router (multiple interfaces active) bridges between transports — for example, relaying a message from a LoRa radio node through TCP to a distant LAN segment.

A Propagation Node is optional but recommended for reliability. It stores messages for nodes that are temporarily offline and delivers them when they reconnect — functioning as an asynchronous store-and-forward relay.

Mesh Network · Hub + Propagation Node
  • Nodes route across multiple hops automatically — no manual routing configuration
  • RF-only nodes (no TCP/LAN) can still participate fully via a hub that bridges to the wider mesh
  • The propagation node stores messages for offline members and delivers them on reconnect
  • A propagation node should run on an always-on machine — a desktop, mini-PC, or single-board computer works well
08 — Messaging

Group Chat

⚠ GhostNet Software Required The Group Chat feature requires all members to run GhostNet software. Standard Reticulum RNS identities — nodes running plain RNS or other LXMF clients — will not be able to participate in GhostNet group chats.

One member's computer functions as the group administrator. Their node acts as the relay server — all messages flow through the admin's station and are distributed to every other member. All other participants are members.

It is recommended that the admin station be a reliable, always-on machine. Running a Propagation Node interface on the admin station gives members the best chance of retrieving messages they missed while offline.

Setup Procedure

  1. Identify members — go to the Network tab in Contacts. Any GhostNet station that has been heard on the mesh will appear here. Check the box next to each station you want to add to your Phone Book.
  2. Confirm they are in the Phone Book — members must be in your Phone Book tab before they can be added to a group. If a station is not in your Phone Book, or is not running GhostNet software, they cannot join a group chat.
  3. Create the group — in the Groups tab, create a new group and add members from your Phone Book.
  4. Announce the group — after the group is created, click the Admin button in the Groups tab and announce the group so members receive the invitation and can join.
ℹ RF Stations & Offline Messages RF-only stations (no TCP or LAN connection) will have difficulty retrieving offline messages. Message retrieval requires an active link back to the relay or propagation node, and the link negotiation process is more demanding than typical message delivery. For reliable offline delivery, the admin station should run a Propagation Node and have a TCP or LAN interface available.
Group Chat · Relay Distribution Model
  • Any member can send to the group — their message is delivered to the relay first
  • The relay distributes to all active members in real time
  • Members who are offline receive queued messages via the propagation node when they reconnect
  • The relay node does not need to be a dedicated machine — any member's node can serve as relay
⚠ Relay Availability If the relay node goes offline, real-time group delivery is interrupted. For critical comms groups, run the relay on an always-on machine or pair it with a propagation node for store-and-forward resilience.
